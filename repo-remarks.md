# Repo Remarks

Scope: readability, small-logic optimization, naming, duplicate structure, and future-agent crawl speed.

This file is for observations that look worth addressing but are too broad, product-sensitive, UX-sensitive, or risky to change during a narrow cleanup pass.

## Open Remarks

- Broad repo optimization is not a single atomic change. Keep future passes split by ownership boundary: shared chat/session logic, wallet/provider logic, web route guards, iOS UI surfaces, bot runtime/admin tooling, and docs/guidelines.
- `apps/bot/src/runtime.js` is doing too much at once: runtime lease/heartbeat, action queue handling, bot session lifecycle, wallet mirroring, chat mirroring, traffic testing, read receipts, and error classification. A future pass should split this by role before adding more bot behavior.
- `apps/ios/app/(vault)/(app)/(home)/camera.js` combines camera device/lens control, QR scanning, capture gestures, preview timing, and transfer/share routing in one large surface. That should be compartmentalized, but doing it safely needs device testing and UX approval.
- `apps/web/src/app/(authenticated)/(vault)/(app)/camera/page.js` mirrors the same risk on web: QR scanning, media recording, upload routing, wallet transfer routing, and browser capability handling all live in one route. Split by scanner, recorder, and routing intent when camera behavior is next touched.
- `apps/web/src/components/wallet/walletdashboard.js` mixes range selection, balance aggregation, peer lookup, chart preparation, and transaction list rendering. A future wallet pass should move chart/range data prep into shared helpers before altering the dashboard UI.
- `apps/ios/app/(vault)/(app)/(home)/settings.js` is a broad settings search/render surface with cache management, retention entry points, legal/review routes, security controls, and logout/delete actions. Split settings row descriptors from action handlers before adding more settings.
- `apps/ios/src/components/media/mediaviewer.js` owns gesture classification, rail math, video/audio controls, save actions, chrome timing, and rendering. It is a good candidate for splitting gesture math and media chrome into focused modules.
- `apps/ios/src/lib/chat/media.js` still combines Firebase Storage upload transport, native AES file crypto, asset sniffing/prep, message upload, and file reads. It now reuses shared filename/filetype helpers, but a full split should wait for native media upload/download testing.
- The web and iOS chat message lists are both large and behavior-sensitive. Future work should separate measurement/scroll anchoring from visible message derivation and per-row interaction glue, but not during a generic cleanup without device/browser testing.
- The web and iOS message row components are still broad render/action surfaces. A future chat UI pass should split row chrome, per-message actions, status labels, and reaction controls only alongside visual regression checks.
- `apps/ios/src/components/avatar.js` is doing more than avatar rendering: color derivation, fallback text, press behavior, bot/active/selected states, upload overlay, and image chrome all meet there. Split visual state helpers only when the avatar system changes next.
- `shared/cache/localdata.js` is the central cache contract and is large because it owns schema, crypto envelopes, media indexes, pruning, and revival. It should probably become a small folder with explicit schema, crypto, media, and prune modules once cache behavior stabilizes.
- The platform cache adapters should stay small: web `src/lib/cache/localdata.js` owns IndexedDB storage shape, iOS `src/lib/cache/localdata.js` owns file storage and native media sealing, and shared `openVaultCache` owns cache semantics. Do not split those adapters just to reduce line count.
- `shared/files.js` combines Firebase Storage transport, avatar paths, encrypted chat media upload/download, and report evidence upload. Split storage paths from encrypted media/report/avatar helpers only in an explicit storage pass.
- `shared/utils/display.js` owns a small mixed display-helper bucket: truncation, byte-size labels, emoji sizing, and DOM/SVG id helpers. Split those only if one sub-concept grows enough to justify another file.
- `shared/providers/txdataprovider.js` still combines transfer normalization, peer aggregation, time-series buckets, range caching, and lookup helpers. Date/hour key formatting is now shared, but a future wallet analytics pass should split aggregation from React provider state.
- `functions/lib/regex.js` repeats the username length from `shared/config.js` because Firebase Functions deploy from the `functions/` source directory. A clean single source of truth likely needs generation or a deploy-package layout change, not a direct relative import from `shared`.
- `functions/passkey/login.js`, `functions/passkey/register.js`, and report/action helpers still repeat small request parsing and uid cleanup. Fold that into `functions/lib/passkey.js` or a functions-local request helper during an explicit backend deploy pass.
- `functions/chat/media.js` has functions-local media-stay naming that overlaps shared message media-stay helpers. Rename or split those backend refs during an explicit media-functions deploy pass, not as drive-by cleanup.
- `scripts/dev.mjs` is useful but broad: process orchestration, line filtering, diagnostics, terminal controls, and sleep-resume handling are all in one file. Split only when changing the runner next, because the current script is operational tooling rather than product logic.
- Web and iOS passkey wrappers still own different credential APIs and prompt sequencing. The shared passkey module now owns neutral error mapping, but future passkey flow changes should keep WebAuthn byte conversion and native passkey prompts platform-owned.
- `apps/web/src/components/providers/adminprovider.js` still combines report lists, report detail subscriptions, moderation state, bot lists, bot event subscriptions, peer priming, and admin mutations. A future pass should split reports and bots into separate provider modules before the admin surface grows.
- `apps/web/src/components/dialogs/mainmenu.js` is a large mixed surface: search, command ranking, peer rows, transaction rows, wallet cache display, and virtualization live together. It should be split by row source and command grouping when main menu behavior changes next.
- The web and iOS chat input components both carry command parsing, draft previews, attachment state, send affordances, reply bars, and input sizing. Shared command primitives already exist; future chat-input work should split platform UI chrome from draft/command derivation without forcing identical web/native layout.
- `apps/web/src/components/dropdownmenu.js` is a custom primitive that owns focus restore, outside-click handling, portal positioning, keyboard navigation, item composition, and styling. Split low-level focus/event helpers only when another primitive needs them; avoid turning it into a generic component utility grab bag.
- Chat retention editing is duplicated across web user details and iOS chat settings. A future pass should extract the pending/server retention state machine behind a shared hook, but the UI and navigation pieces should remain platform-owned.
- Account deletion and wallet export flows have parallel web/iOS implementations. The validation and sensitive-operation sequencing may be shareable later, but any split should be done with explicit review because these are destructive/security-sensitive screens.
- iOS `peerselector`, `sendphoto`, and `sharemedia` repeat grid selection, recent/search merging, selected uid sets, and short peer labels. A future UI pass should extract a native peer grid/selection surface, but that should be reviewed because it affects payment and media-send workflows.
- Wallet send/request flows repeat amount-unit cycling, max-balance checks, self-transfer checks, and peer capability checks across web dialogs and iOS transfer/peerselector screens. A future wallet form pass should extract pure validation/math first, then leave platform UI controls local.
- `shared/chat/usemessages.js` and `shared/chat/messages/session/index.js` are central and large. The next safe compartmentalization would be separating subscription/window state, cache hydration, warm queues, and preview-drop notifications behind smaller helpers with the same public hook boundary.
- `shared/chat/actions/send.js` still owns message composition, local optimistic rows, media permanence, upload adoption, and parent chat patching. A future chat-send pass should split pure message shaping from React callback orchestration.
- `shared/chat/actions/delete.js`, `shared/chat/messages/write.js`, and `shared/chat/usereactions.js` each carry multiple chat lifecycle responsibilities. They should be split only in focused chat passes because small mistakes there affect encrypted retention, receipts, reactions, and parent chat state.
- `shared/providers/chatprovider.js` is the cross-platform chat command center: chat list state, selected-chat memory, message actions, local cache, deletion, retention, and saved media all converge there. Split only after the lower-level chat action modules are stable.
- `shared/providers/userprovider.js` combines auth snapshots, avatar cache sync, ban refresh timers, settings, onboarding fields, and account state. A future auth/profile pass should split identity, avatar, moderation, and settings concerns without moving app routing into the provider.
- The vault providers on web and iOS are still the place where unlock state, key material, route guards, local cache, and boot sequencing meet. Keep route decisions guarded there, but split crypto/cache boot helpers before adding more vault state.
- `shared/providers/peerprovider.js` has grown into the cross-platform peer cache/listener/search boundary. A future peer pass should separate Firestore subscriptions, memory/cache reconciliation, and lookup helpers while preserving one public provider surface.
- `shared/peers.js` combines Firestore profile reads, wallet/chat key lookup indexes, avatar URL resolution, cache merging, and role-aware search support. The next peer pass should split lookup adapters from cache/avatar state.
- `shared/chat/messages/control.js` carries read receipts, hidden checkpoints, reaction grouping, message retention timelines, and seen/hidden filtering. Those are strongly coupled to encrypted chat semantics; split by lifecycle only when changing retention or read-state behavior with focused chat verification.
- `shared/chat/messages/query.js` is compact but dense: Firestore windows, TTL cleanup, auto-hidden deletion, compaction, decrypted-message normalization, and saved-media stay collection are all in one query module. It should be split by lifecycle side effect when chat retention is next changed.
- `shared/wallet/fees.js` is large because unilateral-exit package math, fee-rate labels, and on-chain preview formatting live together. Keep fee formulas local to the wallet fee domain unless a wallet-fee test pass is added; avoid hiding protocol-ish math inside generic number helpers.
- `scripts/repo.mjs` and the admin scripts repeat CLI command shapes (`main`, `usage`, `run`, small parsers) because they are operational entrypoints. Extract only shared argument parsing or output helpers when modifying the scripts together, not as a drive-by abstraction.
- `scripts/web.mjs` and `scripts/ios.mjs` repeat app/network/flag parsing in parallel. A future launcher pass should extract shared script argument parsing, but keep that separate from product cleanup because these scripts are operational entrypoints.
- `scripts/make.mjs` is an operational deploy/build coordinator with generated config, warning summarization, logs, Firebase deploys, app build steps, and prompt handling in one file. It should be split only when changing deployment workflow.
- `apps/ios/app.config.js` still has its own variant alias normalization because Expo evaluates it as CommonJS config. `shared/variant.js` owns runtime normalization now; a future config pass should decide whether to generate config data or move the Expo config to an ESM-safe path.
